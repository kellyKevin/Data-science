## How I Completed the Task from Start to Finish

### Starting Point
When I began this task, I first reviewed the assignment requirements carefully to understand the deliverables: code fixes, data cleaning, candidate ranking, a memo, and a reflection.

### Step 1: Inspect the Data
I opened the provided files (`intern_applications.csv` and `candidates.json`) and examined the headings and sample records. I noted missing values, inconsistent labels, duplicate entries, invalid scores, and any anomalies in the data.

### Step 2: Fix the Broken Script
For the debugging task, I created a fixed version of the analysis script. I improved it by:
- adding error handling for file loading
- replacing `dropna()` with smarter cleaning logic
- validating required columns and values
- checking for unreasonable data such as scores above 100
- summarizing record counts before and after cleaning

This ensured the script ran correctly and produced useful, reliable output.

### Step 3: Clean the Intern Applications Data
I then wrote a dedicated analysis script for the CSV file. The cleaning process included:
- removing duplicate candidate rows
- standardizing track and gender labels
- marking invalid universities and broken emails
- flagging impossible values like a 999 test score and negative days since application
- filling missing numeric values using the median for stability
- validating the cleaned dataset with sanity checks

I saved the cleaned dataset as `cleaned_intern_applications.csv` and generated summary insights.

### Step 4: Build the Ranking Logic
For the candidate ranking task, I developed a transparent scoring system that weights:
- test score (40%)
- project score (35%)
- portfolio score (20%)

I also added penalties for missing documents and bonuses for recent applications. This method is simple, explainable, and easy to discuss during orientation or hiring review.

### Step 5: Write the Memo and Reflection
I prepared a short memo for a non-technical audience explaining why historical model performance is not enough to trust a hiring workflow. I also wrote a reflection answering the required questions about effort, decisions, mistakes, and whether AI tools were used.

## Reasoning Behind My Decisions

### Why I Used Median Imputation
I used the median to fill missing test or project scores because it is robust to outliers. This is important when the dataset contains invalid entries like a score of 999.

### Why I Standardized Categories
I standardized track names and gender values so the results are consistent. Without this step, analysis would split the same category into multiple meaningless groups.

### Why the Ranking Is Transparent
I built a score formula that is easy to explain: it shows how each candidate’s test, project, and portfolio strengths contribute to their overall ranking. This is better than a black-box model for early review and orientation conversations.

### Why I Flagged Data Issues
I flagged candidates with missing or invalid data because those records deserve manual attention. If the data is not clean, it is unfair to rely on the ranking alone.
